## 工作流

1. 直接分析代码 Diff，识别核心改动点。 
2. 快速判断是否需要分组：
- **单次提交**：变更文件属于同一功能模块，不涉及多层级
- **分组提交**：满足以下任一条件
  - 涉及多个独立功能点
  - 跨越 2 个以上系统层级（后端/前端/文档/配置）
3. （若需分组）进行合理分组，详见分组策略 
4. 对每个分组独立或单个提交进行参考代码注释或文档交叉验证。**注意，绝对禁止只看注释或文档就下结论**
5. 只有交叉验证通过后，开始严格按照原则编写Commit消息，若需分组，每组必须独立编写
6. git add 指定的第一组文件，然后在对话中给出第一组的Commit消息（包裹在代码块中方便直接复制），禁止自动提交
7. 等待用户手动提交并明确回复继续，然后开始下一组的第6步重复，如此循环直到所有组完成提交

## 给用户的标准格式（内容仅供参考）

```
feat(segmentation): 实现 Pyannote 软切重构与跨 speaker 残留修复

- 新增 soft_cut 模块（决策引擎、证据构建器、证据融合器），支持基于 speaker 变化事实的延迟决策与跨 chunk 状态维护
- 新增 WordBoundaryMapper 统一"时间点→词边界"映射策略，避免词内事件默认偏向词尾导致延后切分
```

错误示例：

```
git commit -m "build(deps): 升级 ORT 到 1.23.2" -m "- 将 dependencies 中 onnxruntime-gpu 从 1.18.0 升级到 1.23.2
- 将 tool.uv.override-dependencies 中 onnxruntime 从 1.18.0 升级到 1.23.2，保持 CPU 与 GPU 包版本一致
- 选择 1.23.2 作为 Python 3.10 可安装的最新 1.x 版本，避免 1.24.x 缺少 cp310 wheel 导致安装失败"
```

**必须是纯信息**

## 主要原则

1. 提交信息必须使用简体中文
2. 提交格式必须符合 Conventional Commits 规范，每个要点之前添加"- "，具体要求参考Conventional Commits 规范
3. 所有项都不使用"."或者"。"结尾
4. 标题保持精简，不超过 36 字符
5. 正文Body必须严格遵守以下“反废话”原则:

```
1.  **禁止空洞的形容词**：严禁单独使用“提高了用户体验”、“增强了健壮性”、“优化了性能”、“代码更清晰”等主观评价词汇。
2.  **强制技术因果**：如果你想表达“修复”或“优化”，必须说明具体的**技术手段**或**具体现象**。
    * 错误：优化了数据库查询，提升了性能。
    * 正确（简洁版）：为 `user_id` 字段添加索引。
    * 正确（详细版）：为 `user_id` 字段添加索引，将查询时间从 500ms 降低至 20ms。
3.  **要么如实陈述，要么闭嘴**：
    * 如果改动很简单，只陈述改动本身（做了什么），不要强行升华意义。
    * 如果改动很复杂，详细说明“为了解决什么具体问题（Why）”以及“通过什么机制解决的（How）”。
4.  **客观陈述**：只描述代码层面的客观事实，不要包含对未来的推测或情绪化的表达。

请参考以下**正面**与**负面**案例进行生成：
** 负面案例（禁止模仿 - 废话连篇）：**
- feat: 增加了重试机制，提高了系统的稳定性和健壮性，优化了用户体验。
- fix: 修改了空指针报错，增强了代码质量。

** 正面案例 - 简洁风格（推荐）：**
- feat: 在 API 请求失败时增加 3 次指数退避重试。
- fix: 增加对 `response` 对象为 null 的空值校验。

** 正面案例 - 详细风格（仅在需要解释机制时使用）：**
- feat: 在 API 请求失败时增加 3 次指数退避重试。
  - 之前的同步重试会导致高并发下的雪崩效应，改为指数退避以缓解下游服务压力。
```

5. 正文body中一般忽略文档修改，除非这次commit只做了文档修改
6. 正文body中的要点必须按照重要性排序。多个要点时，按以下优先级排列：
   - 核心功能变更（新增/删除/重构的主要能力）
   - 关键机制调整（影响数据流、状态管理的改动）
   - 辅助性改动（工具方法、类型定义、边界处理）
   - 兼容性/防御性代码（向后兼容、异常保护）
7. 脚注footer一般不使用，除非必须使用来明确表示相关重要信息
8. 禁止在Commit消息中使用emoji等图标
9. **绝对禁止在commit中写入类似下面的标识，必须保证消息的纯粹性：**

```
🤖 Generated with [Claude Code](https://claude.com/claude-code)
Co-Authored-By: Claude <noreply@anthropic.com>
```

## Conventional Commits 规范

1. 每个提交都**必须**使用类型字段前缀，它由一个名词构成，诸如 `feat` 或 `fix` ， 其后接**可选的**范围字段，**可选的** `!`，以及**必要的**冒号（英文半角）和空格。
2. 当一个提交为应用或类库实现了新功能时，**必须**使用 `feat` 类型。
3. 当一个提交为应用修复了 bug 时，**必须**使用 `fix` 类型。
4. 范围字段**可以**跟随在类型字段后面。范围**必须**是一个描述某部分代码的名词，并用圆括号包围，例如： `fix(parser):`
5. 描述字段**必须**直接跟在 <类型>(范围) 前缀的冒号和空格之后。 描述指的是对代码变更的简短总结，例如： *fix: array parsing issue when multiple spaces were contained in string* 。
6. 在简短描述之后，**可以**编写较长的提交正文，为代码变更提供额外的上下文信息。正文**必须**起始于描述字段结束的一个空行后。
7. 提交的正文内容自由编写，并**可以**使用空行分隔不同段落。
8. 在正文结束的一个空行之后，**可以**编写一行或多行脚注。每行脚注都**必须**包含 一个令牌（token），后面紧跟 `:<space>` 或 `<space>#` 作为分隔符，后面再紧跟令牌的值（受 [git trailer convention](https://git-scm.com/docs/git-interpret-trailers) 启发）。
9. 脚注的令牌**必须**使用 `-` 作为连字符，比如 `Acked-by` (这样有助于 区分脚注和多行正文)。有一种例外情况就是 `BREAKING CHANGE`，它**可以**被认为是一个令牌。
10. 脚注的值**可以**包含空格和换行，值的解析过程**必须**直到下一个脚注的令牌/分隔符出现为止。
11. 破坏性变更**必须**在提交信息中标记出来，要么在 <类型>(范围) 前缀中标记，要么作为脚注的一项。
12. 包含在脚注中时，破坏性变更**必须**包含大写的文本 `BREAKING CHANGE`，后面紧跟着冒号、空格，然后是描述，例如： *BREAKING CHANGE: environment variables now take precedence over config files* 。
13. 包含在 <类型>(范围) 前缀时，破坏性变更**必须**通过把 `!` 直接放在 `:` 前面标记出来。 如果使用了 `!`，那么脚注中**可以**不写 `BREAKING CHANGE:`， 同时提交信息的描述中**应该**用来描述破坏性变更。
14. 在提交说明中，**可以**使用 `feat` 和 `fix` 之外的类型，比如：*docs: updated ref docs.* 。
15. 工具的实现必须**不区分**大小写地解析构成约定式提交的信息单元，只有 `BREAKING CHANGE` **必须**是大写的。
16. BREAKING-CHANGE 作为脚注的令牌时**必须**是 BREAKING CHANGE 的同义词。

## 分组策略
1. 按以下优先级对文件进行归类，**每个文件只能归入一个分组**：

| 优先级 | 分组类型 | 典型路径特征 | 提交顺序 |
|--------|----------|--------------|----------|
| 1 | 基础设施/配置 | `*.config.*`, `package.json`, `requirements.txt`, `.env*` | 最先 |
| 2 | 数据模型/Schema | `models/`, `schemas/`, `types/`, `*.d.ts` | ↓ |
| 3 | 后端核心逻辑 | `services/`, `utils/`, `core/`, `lib/` | ↓ |
| 4 | 后端接口层 | `routes/`, `api/`, `controllers/`, `handlers/` | ↓ |
| 5 | 前端状态/服务 | `stores/`, `composables/`, `hooks/`, `services/` (前端) | ↓ |
| 6 | 前端组件/视图 | `components/`, `views/`, `pages/` | ↓ |
| 7 | 静态资源/样式 | `assets/`, `styles/`, `public/` | ↓ |
| 8 | 文档/测试 | `docs/`, `*.md`, `tests/`, `__tests__/` | 最后 |

2. 在层级分组基础上，进一步按**功能模块**细分：

- 同一功能的前后端改动可以合并（若改动量小且强耦合）
- 不同功能的同层改动必须分开

构建一个基于 Vue 3、Tailwind CSS 和 Element Plus 的极简 TTS (Text-to-Speech) 界面，核心在于界面的逻辑分层和交互的流畅性。针对你的需求，我们可以采用**左右分栏**或**模块化卡片**的布局方式，以保证界面的整洁。

以下是具体的界面设计方案以及需要预留的后端 API 接口设计。

### 一、 前端界面设计方案

整体框架建议使用一个极简的顶部导航栏或侧边栏配合路由 (`vue-router`)，用于在“模型管理”和“语音合成”两个视图间切换。

#### 1. 模型导入界面 (Model Manager)
此界面主要负责模型的上传与本地列表查看。

* **布局与组件**:
    * **拖拽上传区**: 使用 Element Plus 的 `el-upload` 组件，开启 `drag` 属性。配合 Tailwind 设置一个较大的虚线边框区域，居中显示上传图标和提示文字。
    * **文件管理器触发**: 在拖拽区内放置一个明显的 `el-button`，点击触发系统的文件管理器选择模型文件（通常是 `.pth`, `.pt`, `.onnx` 等格式）。
    * **已导入模型列表**: 上传区域下方，使用 `el-table` 或卡片列表展示当前已存在的模型，包含模型名称、大小、导入时间以及删除按钮。

#### 2. 推理界面 (Inference UI)
这是核心工作区，信息密度较高，建议采用**左右分栏（Grid 或 Flex 布局）**。左侧为配置区，右侧为文本与输出区。

* **左侧：参数与参考配置区 (Config Panel)**
    * **模型选择**: `el-select`，下拉列表数据从后端动态获取。
    * **参考音频配置**:
        * **预设选择**: `el-select`，用于快速选择已内置好的参考音频及对应文本。
        * **自定义参考音频**: 提供一个小型的 `el-upload` 或录音组件，允许用户覆盖预设。
        * **参考音频文本**: `el-input` (type="textarea"，行数设为 2)，显示预设的参考文本，或允许用户手动输入/修改当前参考音频对应的文本内容。
    * **模型参数配置**:
        * 使用 Element Plus 的 `el-collapse` 或简单的表单项排列。
        * 包含常见的 TTS 参数，如语速 (Speed)、语调 (Pitch)、发音人情感/温度 (Temperature/Top_P 等)。使用 `el-slider` 配合 `el-input-number` 实现精准调节。
* **右侧：主输入与输出区 (Input & Output Panel)**
    * **主文本输入**: 一个大尺寸的 `el-input` (type="textarea"，行数设为 8-10)，占据右上部分。
    * **操作区**: 放置显眼的“开始推理”按钮 (`el-button` type="primary")。可以附带一个全局的 `el-loading` 状态指示。
    * **音频预览与历史队列 (Audio Queue)**:
        * 位于主文本框下方。
        * **逻辑实现**: 在 Vue 3 中维护一个响应式数组 `const audioHistory = ref([])`。每次推理完成，将新的音频数据和原始文本推入数组头部 (`unshift`)，并截断数组保持最大长度为 5 (`audioHistory.value = audioHistory.value.slice(0, 5)`)。
        * **UI 呈现**: 使用 `v-for` 渲染这 5 个格子的列表。第一格（最新）可以使用 Tailwind 进行高亮（例如加粗边框或微小的阴影背景区分）。
        * **单格组件设计**: 每个格子包含：对应的合成文本（单行截断省略号）、`<audio>` 原生标签或经过封装的播放器控件、以及一个“下载”按钮 (`el-button` 带有下载图标)。

### 二、 预留后端接口设计 (RESTful API)

为了支撑上述前端功能，后端需要提供以下几类接口。建议数据交互格式统一使用 JSON。

#### 1. 模型管理接口

* **获取模型列表**
    * `GET /api/v1/models`
    * **Response**: 返回可用模型列表，包含 `id`, `name`, `path`, `size` 等字段。
* **上传模型文件**
    * `POST /api/v1/models/upload`
    * **Content-Type**: `multipart/form-data`
    * **Payload**: `file` (模型文件)
    * **Response**: 返回上传状态及分配的模型 `id`。
* **删除模型**
    * `DELETE /api/v1/models/:id`

#### 2. 参考预设接口

* **获取参考音频预设列表**
    * `GET /api/v1/presets`
    * **Response**: 返回预设数组，每项包含 `preset_id`, `name`, `audio_url`, `reference_text`。

#### 3. 核心推理接口

这是整个系统的关键接口，考虑到 TTS 推理可能耗时，根据后端架构可以选择**同步返回**、**异步轮询**或**流式返回 (WebSocket / SSE)**。为了极简起见，这里提供标准的同步阻塞接口方案。

* **执行文本到语音推理**
    * `POST /api/v1/tts/infer`
    * **Content-Type**: `application/json` 或 `multipart/form-data` (如果允许前端直接上传自定义参考音频文件，则必须用 form-data)。
    * **Payload (JSON 示例)**:
        ```json
        {
          "model_id": "string",
          "target_text": "string",
          "reference_preset_id": "string", // 如果使用预设
          "reference_text": "string",      // 参考音频对应的文本
          "parameters": {
            "speed": 1.0,
            "temperature": 0.7,
            "top_p": 0.8
          }
        }
        ```
    * **Response**: 
        * 方案 A: 直接返回二进制音频流 (`Content-Type: audio/wav`)。前端通过 `URL.createObjectURL(blob)` 渲染播放器。
        * 方案 B: 返回 JSON，包含生成的音频文件下载/预览 URL。
        ```json
        {
          "status": "success",
          "audio_url": "/api/v1/audio/download/xyz123.wav",
          "duration": 5.2
        }
        ```

#### 4. 音频获取/下载接口 (若推理接口使用方案 B)

* **获取/下载生成的音频**
    * `GET /api/v1/audio/download/:filename`
    * **Response**: 音频二进制流，可配置 `Content-Disposition: attachment` 头以强制触发浏览器下载行为。

### 三、 关键实现细节与建议

1.  **状态管理**: 由于应用较为轻量，直接利用 Vue 3 的 Composition API 和 `ref/reactive` 即可在父子组件间进行状态传递，暂时不需要引入 Pinia，除非后续要加入复杂的用户配置持久化。
2.  **防抖与防重**: “开始推理”按钮在点击后必须立即置为 `disabled` 状态，并开启 Loading 动画，直到后端返回结果或超时报错，防止用户重复点击导致显存溢出或队列阻塞。
3.  **音频释放**: 对于前端使用 `URL.createObjectURL` 生成的本地 Blob 链接，由于维护了一个长度为 5 的队列，当有旧音频被挤出队列时，建议在 Vue 的生命周期或侦听器中调用 `URL.revokeObjectURL(oldUrl)`，以避免内存泄漏。
